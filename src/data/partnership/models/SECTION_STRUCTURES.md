### 1. `meta`

**Present in:** All 16 models  
**Optional:** No

**TypeScript interface (union of all fields):**

```ts
interface Meta {
  // Always present
  slug: string;
  system: 'core-trade' | 'brand-growth' | 'collaborative' | 'institutional';
  tier: 1 | 2 | 3 | 4;
  label: string;
  pageTitle: string;
  seoDescription: string;
  kpiRequired: string;           // e.g. '7.5+', '8.0+', '9.0+'
  retentionRequired: string;     // e.g. 'None', '12 months at Warehousing'
  previousSlug: string | null;
  nextSlug: string | null;
  lateralEntryAvailable: boolean;
  fastTrackAvailable: boolean;
  tags: readonly string[];

  // Optional — cross-system progression fields
  nextSystem?: string;           // present on: ImportExport, BrickMortar, Warehouse
  previousSystem?: string;       // present on: Exhibition, Packaging (and all Collaborative/Institutional models)

  // Optional — terminal marker
  isTerminal?: true;             // present on: TechnologyPartnership only
}
```

**Notes:**
- `previousSlug: null` only on the first model in each system (e.g., Dropshipping has `previousSlug: null`)
- `nextSlug: null` only on TechnologyPartnership (`isTerminal: true`)
- `system` values: `'core-trade'` (Tier 1), `'brand-growth'` (Tier 2), `'collaborative'` (Tier 3), `'institutional'` (Tier 4)
- `kpiRequired` escalates across tiers: `'7.0+'` → `'7.5+'` → `'8.0+'` → `'8.5+'` → `'9.0+'`

### 6. `eligibilityAssessmentSection`

**Present in:** All 16 models  
**Optional:** No

**TypeScript interface:**

```ts
type AssessmentCategory = 'general' | 'validation' | 'vendor' | 'buyer';
type QuestionType = 'single' | 'multiple' | 'yesno';

interface AssessmentQuestion {
  id: string;
  question: string;
  type: QuestionType;
  options: readonly string[];
  category: AssessmentCategory;
  weight: 0 | 1;        // scored (1) or unscored (0)
}

interface AssessmentThreshold {
  eligibility: string;            // must match resultContent keys exactly
  fastTrackEligible: boolean;
  nextLevel: string | null;
}

interface SpecialRulesVendor {
  certificationsRequired: boolean;
  commitment: string;
  kpiThreshold: number;

  // Optional — present on higher-tier models
  retentionPeriod?: string;       // present from Consignment upward
  minCommitment?: string;         // present from Packaging upward
  tpif?: string;                  // present on: Museum, NGO & Government, TechnologyPartnership
  mpf?: string;                   // present on: Museum, NGO & Government, TechnologyPartnership
}

interface SpecialRulesBuyer {
  commitment: string;
  kpiThreshold: number;

  // Optional — present on institutional models
  cep?: string;                   // present on: Museum, NGO & Government, TechnologyPartnership ('Open Access')
}

interface ResultContent {
  messagesByOutcome: Record<string, string>;
  nextStepsByOutcome: Record<string, string[]>;
  strengthsByOutcome: Record<string, string[]>;
  improvementsByOutcome: Record<string, string[]>;
  defaultStrengths: string[];
  defaultDevelopmentAreas: string[];
}

interface EligibilityAssessmentSection {
  sectionLabel: string;
  headline: string;
  description: string;

  introCard: {
    icon: string;
    title: string;
    description: string;
    primaryCta: { label: string; action: 'start-assessment' };
    meta: string;            // e.g. '10 questions • Personalised results • No email required'
  };

  resultCard: {
    heading: string;
    scoreLabel: string;
    recommendationLabel: string;
    strengthsLabel: string;
    developmentLabel: string;
    nextStepsLabel: string;
    retakeCtaLabel: string;
    fastTrackBadgeLabel: string;
  };

  assessment: {
    name: string;
    maxScore: 25;             // constant across all models
    questions: readonly AssessmentQuestion[];
    thresholds: {
      10: AssessmentThreshold;
      16: AssessmentThreshold;
      21: AssessmentThreshold;
    };
    specialRules: {
      vendor: SpecialRulesVendor;
      buyer: SpecialRulesBuyer;
    };
    resultContent: ResultContent;
  };
}
```

**Role-aware fields:** Questions are filtered by `category` at runtime (`'vendor'` questions shown only to vendors, `'buyer'` questions shown only to buyers). `general` and `validation` questions apply to all roles.

**Universal fields:** `sectionLabel`, `headline`, `description`, `introCard`, `resultCard`, `assessment.name`, `assessment.maxScore`, `assessment.thresholds`, `assessment.specialRules`, `assessment.resultContent`

**Notes on `validation` category:**
- `'validation'` category questions are ABSENT in: Dropshipping, Museum, NGO & Government, TechnologyPartnership
- These models have only `'general' | 'vendor' | 'buyer'` categories
- Museum, NGO & Government, TechnologyPartnership have no `validation` category because their entry has no prior retention requirement
- Dropshipping has no `validation` category because it is the first-tier entry point

**Notes on question counts:**
- 15 questions: Logistics (most comprehensive, 5 general + 4 validation + 3 vendor + 3 buyer)
- 13 questions: most Collaborative models
- 11 questions: Museum, NGO & Government (no validation category — 5 general + 3 vendor + 3 buyer)
- 12 questions: TechnologyPartnership (5 general + 3 vendor + 4 buyer — buyer has one extra)
- 10 questions: Dropshipping (no validation, 5 general + 3 vendor + 2 buyer)

**Notes on `thresholds` keys:** Always exactly `10`, `16`, `21`. Keys are numeric. The `eligibility` string values must match resultContent keys exactly — they are used as lookup keys by the EligibilityProvider component.

**Notes on `specialRules.vendor`:**
- Tier 1 models (Dropshipping): only `certificationsRequired`, `commitment`, `kpiThreshold`
- Tier 1 (Consignment) onwards: adds `retentionPeriod`
- Tier 3 (Packaging) onwards: adds `minCommitment`
- Institutional parallel tracks (Museum, NGO, Technology): adds `tpif` and `mpf`

## Cross-Cutting Patterns

### Role Content Architecture

Every section that varies by role uses one of these patterns:

**Pattern A — Full role split** (heroSection, modelDifferenceSection, platformAdvantageSection, finalCtaSection):
```ts
{
  shared: { /* always shown */ },
  roles: {
    buyer: { /* buyer-specific content */ },
    vendor: { /* vendor-specific content */ },
  }
}
```

**Pattern B — Universal** (overviewSection, eligibilityRulesSection, capabilitiesSection, pricingSection, technologySection, apiIntegrationSection, workflowSection, faqSection, keyFeaturesSection, comparisonSection):
No role split — identical content shown to all roles.

**Pattern C — Runtime filter** (eligibilityAssessmentSection):
Questions carry a `category` field. The component filters questions at runtime: `general` and `validation` apply to all, `vendor` only to vendors, `buyer` only to buyers.

### Museum / NGO / Technology — Institutional Parallel Tracks

These three Institutional models (Museum, NGO & Government, Technology Partnership) are parallel tracks — not sequential. They share:
- `retentionRequired: 'None — institutional qualification required'` in meta
- `specialRules.vendor.retentionPeriod: 'None'`
- `'Prior Retention': 'None required'` in eligibilityRulesSection requirements
- No `validation` category in assessment questions
- `specialRules.buyer.cep: 'Open Access'`
- Buyer CTA: `'Apply as an Institution'` (Museum, NGO) or `'Apply for Technology Partnership'` (Technology)

Logistics is the sequential gateway that precedes them — it has a `validation` category and requires prior Warehousing retention.
